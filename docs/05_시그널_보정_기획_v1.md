# 시그널 임계값·가중치 보정 기획 (v1)

> 작성일: 2026-07-13 | 선행 문서: 종합기획서 v2.4, 발굴로직 보강 v1(04)
> 배경: 로드맵 Phase 4 "시그널 백테스트(θ·가중치 보정)" 및 04번 문서 미결사항
> ("종목 점수 가중치 0.5/0.5, 적중 임계값 θ·가중치 백테스트 보정")을 구체화한다.
> 코드베이스에 이미 존재하는 모든 하드코딩 가중치·임계값을 실사하고, 검증→보정
> 파이프라인을 설계한다. 이번 문서는 **산업 분류/매핑 정확도, 리포트 파싱 정확도,
> 애널리스트 Top3 자동교체는 다루지 않는다** (별도 기획 대상).

## 0. 현황 진단: 무엇이 검증됐고 무엇이 안 됐는가

| 성분 | 위치 | 현재 값 | 검증 상태 |
|---|---|---|---|
| L1 상대강도(RS)·추세 | `src/analysis/backtest.py` | — | **검증됨**: IC=0.163 (n=420, 6개월치), RS상위군 fwd 20일 +13.5% vs 하위군 -1.3% |

**결론**: 실증 근거가 있는 성분은 L1(RS·Breadth)뿐이고, 나머지는 모두 "그럴듯한 직관값"이다. 다만 데이터는 이미 상당수 쌓여 있어(stock_scores 9일×918행, eps_consensus 8회 스냅샷) 1차 보정에 착수할 여건은 된다.

## 1. 문제 인식

1. **표본 제약**: 가격이력이 ~6~7개월뿐이라 시계열 자유도가 낮고, 산업 간 공통 시장요인 때문에 관측치가 서로 독립이 아니다(표면적 n_obs보다 유효표본이 작음). 지금 정밀 최적화(회귀·그리드서치)를 돌리면 노이즈에 과적합될 위험이 크다.
2. **as-of 무결성**: `report_events.published_date`, `eps_consensus.snapshot`은 발행/수집일이 박혀 있어 as-of 재구성이 가능하지만, `l0_signals`는 이력이 없어 "그 시점 값"을 복원할 수 없다 — 백테스트 이전에 스키마 보완이 선행돼야 한다.
3. **원칙 정합성**: 가중치 보정은 어디까지나 룰 기반 파라미터 조정이어야 한다(CLAUDE.md 원칙 2, LLM 미개입). 회귀/ML 기반 자동 가중치 학습은 표본이 훨씬 쌓인 뒤에나 검토 대상이지, 지금 도입 대상이 아니다.

## 2. 결정: 3단계 보정 파이프라인

### 2.1 1단계 — 성분별 개별 예측력 측정 (지금 바로 착수 가능한 것부터)

`src/analysis/backtest.py`(L1 전용)를 확장하기보다, 성분이 늘어날 것을 고려해 `src/analysis/backtest_components.py`를 신설한다. 각 성분에 대해 IC(Spearman)·버킷 스프레드·n_obs를 계산하고, **최소 표본 미달 시 숫자를 억지로 내지 않고 "판단 보류"로 명시**한다.

착수 순서(데이터 가용성 기준):

| 순서 | 성분 | 전제 데이터 | 현재 가능 여부 |
|---|---|---|---|
| 1 | 확인 레이어 breadth/magnitude/upside_gap | `report_events.published_date` (as-of 안전) | **가능** — 즉시 착수 |
| 2 | 종목 buy_score 구성요소(upside_pct/composite/share_bonus) | `stock_scores`(9일치) + `price_history`(대표종목 122개) | **가능** — 종목별 forward return 신규 계산 필요 |
| 3 | 선행 레이어(EPS 리비전·가속도·최초전환) | `eps_consensus` 스냅샷 시계열 | **부분 가능** — 8회 축적, 가속도(2차 미분)는 최소 3~4주 더 필요 |
| 4 | L0 업황 동인(수출/재고순환/가격) | `l0_signals` 이력 | **불가** — 이력 테이블 부재. 먼저 `l0_signal_history`류 테이블을 추가해 일별 적재로 전환해야 함 |

### 2.2 2단계 — 가중치 재산정 규칙

- 원칙: **"IC가 검증된 성분에 비례 배분"**만 적용한다. 즉 `w_i ∝ max(IC_i, 0)`로 정규화하고, 미검증 성분은 현재 값을 유지(임의로 0으로 만들지 않음 — 검증 안 됐다고 무가치는 아니므로).
- 그리드서치·회귀 기반 최적화는 성분별 유효표본이 충분히 쌓인 뒤(추후 판단 기준은 2.3 참조)에만 검토한다.
- 변경 이력은 `data/output/signal_weights.json`에 버전 기록한다: `{as_of, component, old_weight, new_weight, ic, n_obs}[]`. 대시보드 백테스트 카드에 "이 가중치는 YYYY-MM-DD, IC=0.xx 근거로 조정됨" 배지를 노출할 수 있게 한다.
- 최소 표본 기준(예: n_obs<100 또는 관측기간<3개월) 미달이면 가중치를 동결하고 "데이터 축적 중"으로 표시한다.

### 2.3 3단계 — 임계값 그리드서치 (2단계 안정화 이후)

`cycle_classifier`의 cb 컷오프(0.6 / -0.4), `timing.py`의 RSI 75·breadth 60% 같은 이산 임계값은 가중치와 별개로 후보값(예: 0.4/0.5/0.6/0.7)을 스윕해 forward return 기준 최적점을 찾는다. **2단계가 몇 주 이상 안정적으로 돌아간 뒤에 시작** — 지금 당장 하면 임계값을 노이즈에 맞추는 결과가 된다.

## 3. 실행 순서 제안

1. `backtest_components.py` 신설 — 확인 레이어 IC부터 (표 2.1의 순서 1)
2. 종목 레벨 백테스트 추가 — 순서 2. `stock_scores.calc_date` 기준 as-of composite/upside_pct/share_bonus vs 이후 N거래일 종목 수익률
3. `l0_signals`를 이력 테이블로 전환(스키마 변경: PK를 `(level2_id, calc_date)`로) — 이 작업 없이는 L0 가중치 보정이 원천적으로 불가능하므로 우선순위 높게 별도 처리
4. eps_consensus 스냅샷이 4주 이상 쌓이면 선행 레이어 IC 측정 (순서 3)
5. 1~4 결과를 `data/output/signal_weights.json`으로 발행하고, `cycle_classifier`/`build_static`/`stock_scorer`/`timing`이 하드코딩 상수 대신 이 파일을 읽도록 리팩터 (로직 구조는 유지, 숫자만 외부화)
6. 대시보드 백테스트 카드에 성분별 IC·스프레드·표본수를 노출해 신뢰도를 가시화
