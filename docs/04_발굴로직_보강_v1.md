## 0. 검토 요약: 깔때기의 3개 관절

기획서 1.1의 투자 로직은 **거시 → 산업 선별 → 산업 분석 → 기업/ETF** 4단계다. 이를 신호 관점에서 보면 단계 사이에 3개의 "관절"이 있고, v2.0 설계에서 각 관절의 연결 상태는 다음과 같았다.

| 관절 | v2.0 상태 | 문제 | 본 문서 결정 |
|---|---|---|---|
| ① 거시→산업 | 거시는 "거시 현황" 화면에 표시만, 산업 스코어에 미반영 | 탑다운 1번 단계가 신호로 미구현 | **참고 화면 유지** (의도적 단순화) |
| ② 산업 선별 | 목표가 컨센서스 모멘텀(동행~후행 군중 신호) | "발굴=선행"과 방향이 어긋날 수 있음 | **선행 보강 하이브리드** |
| ③ 산업→종목 | 대표기업 시드 매핑만, 종목 선별 점수 없음 | 발굴이 섹터에서 멈춤 | **펀더멘털 모멘텀 점수화** |

## 1. 관절② — 선행 보강 하이브리드 (산업 선별 신호)

### 1.1 문제 인식

기존 1순위 신호인 "섹터 목표가 동시 상향(Breadth/Magnitude/Upside Gap)"은 **애널리스트 목표가 리비전**에 기반한다. 목표가 리비전은 통상 실적·가이던스 변화에 **후행**하므로, 섹터 전체에 상향이 깔릴 때쯤이면 주가가 이미 상당 부분 반영된 경우가 많다. "후행성 보정"(통합기획서 2.2.1)은 이를 *경고*할 뿐, 더 이른 진입점을 *생성*하지 않는다.

### 1.2 결정: 선행 레이어 3종 추가 (기존 3지표는 유지)

기존 목표가 모멘텀 3지표는 **확인(confirmation) 레이어**로 유지하고, 그 앞단에 **선행(leading) 레이어**를 얹는다.

| 선행 지표 | 정의 | 근거 |
|---|---|---|
| **EPS 리비전 Breadth/Magnitude** | 목표가 대신 **EPS 추정치**에 동일한 확산도·상향폭 공식 적용 | 이익 추정치 리비전이 목표가 리비전을 선행 (애널리스트는 이익 전망을 먼저 고치고 목표가를 뒤따라 조정) |
| **리비전 가속도 (2차 미분)** | 직전 윈도우 대비 리비전율의 변화 = Δ(리비전율) | 절대 리비전이 아니라 *리비전의 가속*이 변곡점을 포착. "이제 막 돌기 시작한" 섹터 탐지 |
| **최초 전환 탐지** | 유니버스 내에서 Breadth가 음→양으로 가장 최근에 교차한 섹터 플래그 | "다같이 오르는 게 확인된" 섹터가 아니라 *가장 먼저 도는* 섹터 = 발굴 우선순위 |

### 1.3 신호 우선순위 (대시보드 정렬)

```
1순위: 최초 전환 + EPS 리비전 가속도 양(+)        → "지금 막 발굴할 섹터"
2순위: 목표가 모멘텀 3지표 확인 (Breadth≥+0.6)    → "이미 확인된 섹터" (확신용)
보정:  후행성/희소섹터 보정은 양 레이어 모두에 적용
```

핵심: **선행 레이어가 발굴(조기 포착), 확인 레이어가 검증.** 둘이 같이 켜지면 신뢰도 최상, 선행만 켜지면 관찰 후보, 확인만 켜지면 추격 주의.

### 1.4 데이터/스키마 영향

- `report_events`에 `eps_estimate`, `prev_eps_est`, `eps_fwd_year` 컬럼 추가 (스키마 반영 완료)
- EPS 추정치 확보 경로: 네이버 리서치 상세/컨센서스 집계. 리포트별 추정치가 없으면 종목 컨센서스 스냅샷으로 폴백 (Phase 2에서 소스 확정)
- 가속도·최초 전환은 시계열에서 **계산**하므로 추가 원시 테이블 불필요 (published_date + 리비전 값으로 산출)

## 2. 관절③ — 펀더멘털 모멘텀 종목 선별 (산업→종목)

### 2.1 문제 인식

섹터 신호가 발동해도("조선 Breadth +0.7"), 그 안에서 *어떤 종목*을 볼지 점수화하는 로직이 없었다. 발굴이 산업 단위에서 멈춰 종목까지 내려오지 못한다.

### 2.2 결정: 펀더멘털 모멘텀 점수 (섹터 내 종목 랭킹)

핫 섹터 안의 종목을 다음 점수로 정렬한다.

```
종목_점수 = z(EPS 추정치 리비전 %)      × 0.5     # 이익 전망이 오르고 있는가
          + z(영업이익률 추세, 2분기 Δ)  × 0.5     # 실제 수익성이 개선되는가
```

- **EPS 추정치 리비전**: `report_events.eps_estimate` / `company_fundamentals.eps_consensus`의 종목별 변화율
- **영업이익률 추세**: `company_fundamentals.op_margin`의 최근 2분기 연속 방향 (통합기획서 2.3 기업 보조 시그널 "영업이익률 2분기 연속 개선" 정량화)
- **(보조) Upside Gap**: 동점 시 목표가-현재가 괴리율로 타이브레이크. 1차 점수엔 미포함 (검토 결과 펀더멘털 우선)

### 2.3 데이터/스키마 영향

- 신규 테이블 `company_fundamentals(ticker, period_end, revenue, op_income, op_margin, net_income, eps_actual, eps_consensus, source)` 추가 (스키마 반영 완료)
- 실적(actual)은 **DART OpenAPI**, 컨센서스는 추정치 집계. Phase 1은 스키마만 확보, 적재는 Phase 2~3
- 산업 귀속은 종목코드 기준(CLAUDE.md 원칙 4) 그대로 — 종목 점수는 이미 Level2에 귀속된 종목들 사이에서만 계산

## 3. 관절① — 거시는 참고 화면으로 유지 (의도적 미연결)

### 3.1 결정

거시 지표(금리·환율·수출·CPI)를 산업 스코어에 **능동 반영하지 않는다.** "거시 현황" 화면으로 추세만 제공하고, 거시 국면 판단은 사용자가 직접 한다.

### 3.2 근거

- 거시→산업 민감도 맵(예: 금리↓→건설/리츠)은 관계가 비선형·국면의존적이라 룰로 고정하면 오신호 위험이 큼
- 1인 사용자가 거시 맥락은 스스로 잘 판단하는 영역 → 자동화 ROI 낮음
- 복잡도를 신호 엔진에서 덜어내 ②③ 발굴 정확도에 집중

### 3.3 영향

- `industry_macro_sensitivity` 류의 테이블/로직 **만들지 않음**
- 거시 수집(ECOS)은 Phase 3 대시보드 화면용으로만 유지. 신호 엔진과 분리

## 4. 종합: 보강된 발굴 파이프라인

```
[거시 현황]  참고 화면 (사용자 판단)        ← 관절① 미연결
     │
     ▼
[산업 선별]  선행 레이어(EPS 리비전·가속도·최초 전환)  → 발굴 후보
            확인 레이어(목표가 Breadth·Magnitude·Upside Gap) → 검증   ← 관절②
     │
     ▼
[종목 선별]  섹터 내 펀더멘털 모멘텀 점수(EPS 리비전 + 영업이익률 추세) → 종목 랭킹   ← 관절③
     │
     ▼
[애널리스트] 해당 종목/산업 Top3 리포트 이벤트 (출처·링크 2종)
```

## 7. 단기 낙폭과대 보조 신호 + 목표주가 컨센서스 절사평균 (v1.1 추가, 2026-07-21)

### 7.1 배경

CLAUDE.md 서비스 목적상 이 서비스는 일간 이슈 트래커가 아니라 **주간 사이클 국면**을 우선한다.
그러나 "그날그날 주가 등락에 따라 매수 타이밍이 좋아질 수 있다(과대 낙폭 등)"는 현실은 무시할 수
없는 보조 정보다. 이를 사이클 판정(`timing_state`)과 분리된 **별도 보조 신호**로 추가한다 —
기존 buy/watch/hold/caution 판정 로직(주간 추세 기반)은 그대로 두고, 헤드라인 신호로 승격하지 않는다.

### 7.2 결정: `oversold_flag` (단기 낙폭과대)

`timing_signals`에 다음 3개 컬럼 추가, 세 조건 동시 충족 시에만 1로 판정(노이즈 최소화):

| 지표 | 임계값 |
|---|---|
| 5거래일(1주) 수익률 | ≤ −7% |
| RSI(14) | ≤ 32 |
| 20일선 대비 이격도 | ≤ −8% |

`timing_score`에는 +8점만 가산(다른 보조 가산 요소와 동일 체급)하고, `timing_state` 자체는 바꾸지
않는다. 대시보드에는 사이클 배지와 별개의 "⚡ 낙폭과대" 배지로 노출 — 사이클이 침체·둔화 상태여도
이 배지 하나로 "매수 적기"처럼 보이지 않도록 한다(원칙②: 판단은 룰 엔진이 하되, 두 신호의 성격이
다르면 뒤섞어 보여주지 않는다).

### 7.3 결정: 산업별 목표주가 컨센서스 절사평균

원화 목표주가는 종목별 주가 스케일이 달라 산업 단위로 원 단위 평균을 낼 수 없다. 따라서:

1. 종목별: 최근 12주 리포트의 목표주가들 중 최고·최저 1개씩 제거 후 평균(3건 미만이면 단순평균).
2. 산업별: 위 종목별 절사평균 목표가 → 현재가 대비 상승여력(%)을 구하고, 산업 내 종목들의
   상승여력 값들 중 최고·최저 1개씩을 다시 제거 후 평균 — "산업별 목표주가 컨센서스(절사평균)".

`stock_scorer._upside`(종목 upside_pct·buy_score 산출에도 동일 절사평균 적용)와
`build_static.py`(`industries.json`의 `target_upside_avg`/`target_upside_n`, 산업 카드·상세 화면에
노출)에 반영.

## 8. 퀀트 검증 결과 및 로직 보강 (2026-07-21)

두 로직을 실제 DB(가격이력 ~6개월·30개 산업, 목표가 리포트 ~2.5개월·965건)로 사후검증하고,
발견된 약점을 즉시 보강했다. 재실행 가능한 백테스트는 `src/analysis/backtest.py`
(`_run_oversold`)에 상시 포함해 대시보드 "🔬 신호 검증" 카드에 노출한다.

### 8.1 `oversold_flag` 검증 — 실제로 forward 수익률을 예측하는가

과거 모든 거래일에 대해 as-of 시점 지표(5일 수익률·RSI14·20일선 이격도)를 룩어헤드 없이
재구성하고, 이후 5·10·20거래일 산업지수 수익률을 baseline(전체 관측치)과 비교했다. 일별
관측치는 같은 급락 구간이 여러 날 연속 플래그되어 자기상관이 크므로, 연속 구간을 1개 "이벤트"로
묶어 최초일 기준으로도 별도 집계(유효 표본에 더 가까움).

| 구간 | 이벤트(n=36, 20개 산업 분산) | baseline(n=2550) | 초과 | Welch t | 유의(5%)? |
|---|---|---|---|---|---|
| forward 5거래일 | +3.14% (승률 63.9%) | +1.21% (승률 53.6%) | +2.0%p | 1.54 | **아니오** |
| forward 10거래일 | +2.26% (승률 63.9%) | +2.24% (승률 50.9%) | +0.0%p | 0.04 | 아니오 |
| forward 20거래일 | −0.58% (승률 50.0%) | +2.76% (승률 46.7%) | −3.4%p | −1.41 | 아니오 |

**최초 검증(위 표만 봤을 때)에서 "1주 초과수익 확인"이라고 결론 내렸던 것은 과대해석이었다** —
사용자 피드백으로 재점검. 평균값 차이만 보고 승률 차이(63.9% vs 53.6%)에 이끌려 유의성 검정을
생략한 게 원인. Welch's t 근사로 재검증한 결과 세 구간 모두 |t|<1.96로 통계적으로 유의하지
않다(가격이력 6개월·단일 상승국면·이벤트 36건이 근본 원인). fwd5일 기준 80% 검정력을 확보하려면
그룹당 ~330건(현재의 약 9배) 필요 추정 — 현재 이벤트 발생 속도(약 6개월에 36건)로는 4년 이상
더 누적해야 하는 규모라, "몇 개월 더 쌓기"로 해결될 문제가 아니다. 임계값 대안 비교(RSI 단독,
조건 완화판 등)도 마찬가지로 표본이 작아 상대적 우열 자체가 노이즈일 가능성이 크다.

**보강(수정)**: `timing_score`에 주던 +8점 가산을 **제거**했다(2026-07-21) — 유의성이 확보되지
않은 신호를 점수(=판단)에 반영하는 것은 CLAUDE.md 원칙②("LLM은 판단하지 않고 설명만") 이전에
"룰 엔진 자체가 근거 없이 판단"하는 문제다. `oversold_flag`는 timing_state·timing_score에
어떤 영향도 주지 않는 **순수 정보성 배지**로만 유지하고, 대시보드·백테스트 카드에 t값·필요표본
크기를 함께 노출해 사용자가 신뢰도를 직접 판단할 수 있게 했다(`src/analysis/backtest.py`
`_stats_test`, `index.html` `renderOversoldBacktest`). 임계값(3조건 AND) 자체는 기술적 분석의
표준적 정의(RSI 과매도·단기 급락·이평 이격)를 그대로 쓴 것이므로 유지하되, "검증된 규칙"이
아니라 "미검증 휴리스틱"이라고 문서·UI 모두에 명시한다.

### 8.2 목표주가 절사평균 검증 — 절사만으로 충분한가

종목별 목표가 분포를 점검한 결과, 절사평균이 단순평균 대비 최대 +39%p까지 값을 바로잡는 사례를
확인해 절사의 효용은 확인했다. 그러나 **실제 데이터에서 절사만으로는 부족한 사례**를 발견:

- 두산밥캣(241560): 리포트 3건 중 2건(200만·222만원)이 종가(6.31만원)의 30배 이상. 나머지
  1건(9.6만원, 1.5배)이 실제로 타당한 값으로 보이나, 절사평균은 최고·최저 각 1개만 제거하므로
  최저값(9.6만원=타당값)이 제거되고 중간값(200만원=이상치)이 그대로 채택되는 역효과 발생.
  **"이상치가 항상 배열의 양끝에 1개씩만 있다"는 절사평균의 전제가 다수(2/3건) 오염 시 깨진다.**
- 효성중공업(298040): 리포트 6건 중 5건이 400만~500만원대로 밀집, 1건만 18.5만원(20배 이상
  괴리) — 이 경우는 절사평균이 정상 작동(최저값 제거로 올바르게 보정).

**보강**: 절사 전에 "목표가/현재가 배율이 0.2~5.0배를 벗어나면 파싱 오류로 간주해 제외"하는
정합성 필터(`_sane_targets`)를 `stock_scorer.py`·`build_static.py`에 추가. 전량 제외되면(예:
전 종목 목표가가 시스템적으로 현재가의 10배 이상인 케이스 — 코스맥스 044820에서 발견, 9건 전부
현재가의 11~13배지만 가격이력 자체는 정상 추세라 목표가 쪽이 의심됨) 과잉 제거를 막기 위해
원본으로 폴백한다. 필터 적용 후 재계산 결과: 두산밥캣은 9.6만원(현실적 값)으로, 효성중공업은
496.7만원(기존 472.5만원보다 이상치 영향이 더 줄어든 값)으로 개선.

### 8.3 데이터 한도로 검증하지 못한 것

- 목표가 컨센서스의 forward 주가예측력(IC) 자체는 리포트 이력이 ~2.5개월뿐이라 통계적으로
  신뢰할 수 있는 백테스트가 불가능해 시도하지 않음(과최적화 위험이 더 큼) — 6개월 이상 누적 후
  재검증 필요.
- `oversold_flag`를 `cycle_signals`/`l0_signals`(사이클 국면)로 조건부 분리(예: 침체 국면
  제외)하면 "낙폭과대이면서 진짜 폭락 중"인 falling-knife를 걸러낼 수 있는지는 `cycle_signals`
  이력이 18개 스냅샷(~1개월)뿐이라 검증 불가 — 데이터 누적 후 재검토.
