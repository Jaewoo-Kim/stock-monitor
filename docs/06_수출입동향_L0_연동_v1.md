# 산자부 수출입동향 → L0 업황 동인 연동 (v1)

> 작성일: 2026-07-13 | 관련 코드: `src/collectors/customs_trade.py`, `src/signals/l0_industry.py`,
> `data/seed/industry_indicators.json`
> 배경: L0 업황 동인 레이어의 "수출" 지표(반도체·이차전지·디스플레이·자동차·선박·석유화학·
> 철강·석유제품 8개 산업)가 `source: "ecos"`로 지정된 채 ECOS stat_code가 한 번도 채워진 적이
> 없어 **한 건도 수집되지 않고 있었다** (`industry_indicators` 테이블 0행). 이 문서는 이를
> 산업통상자원부가 매월 발표하는 수출입동향과 동일한 품목 분류를 사용하는 **관세청 무역통계
> 오픈API**로 메우는 작업의 결정과 검증 절차를 기록한다.

## 1. 데이터 소스 선택

세 가지 무료 경로를 검토했다 (사용자 결정: 관세청 API):

| 경로 | 특성 | 채택 여부 |
|---|---|---|
| 관세청 무역통계 오픈API (data.go.kr, `관세청_품목별 수출입실적(GW)`) | HS코드 기준 REST API, 과거 시계열 조회 가능(YoY 계산에 필요한 13개월+ 확보 용이) | **채택** |
| 산자부 월간 보도자료(motie.go.kr) 파싱 | 산업 분류가 기존 taxonomy와 정확히 일치, 그러나 PDF/HWP라 파싱 취약 + 과거 이력 없음(지금부터 축적해야 함) | 미채택 (파싱 취약성이 이미 CLAUDE.md가 지적한 리포트 크롤러의 문제와 동일) |
| 공공데이터포털 파일데이터(`3039964`) 다운로드 | 구조화 파일이라 파싱은 쉬우나, 실제 갱신 주기·컬럼 구성을 이 세션에서 검증 못함 | 미채택 |

## 2. 이 세션의 제약: 네트워크 접근 불가

이 환경의 아웃바운드 프록시가 `data.go.kr`/`tradedata.go.kr` 등 정부 사이트로의 직접 연결을
정책상 차단한다(`CONNECT` 403). 따라서 실제 API를 호출해 다음을 **검증하지 못한 채** 코드를
작성했다:

- 정확한 엔드포인트 경로 (현재 `apis.data.go.kr/1220000/nitemtrade/getNitemtradeList`로 추정)
- 요청 파라미터명 (현재 `strtYymm`/`endYymm`/`hsSgn`로 추정 — data.go.kr 관세청 API 계열의
  일반적 명명 관례를 따른 것이며 이 API에 정확히 맞는지는 미확인)
- 응답 필드명 (현재 `expDlrAmt` 등 후보를 시도하고, 안 맞으면 로그에 실제 키 목록을 남기도록
  방어적으로 작성)
- HS코드 매핑의 정확성 (아래 3장)

**따라서 `industry_indicators.json`의 각 export 지표에 `customs.verified: false`를 기본값으로
두었고, 수집기는 `verified: true`인 지표만 수집한다.** 틀린 코드로 잘못된 수치가 조용히
신호 엔진에 들어가는 것을 막기 위함이다(CLAUDE.md 원칙 3 정신 — 출처·정확도 우선).

## 4. 검증 절차 (네트워크 접근 가능한 세션/사용자가 수행)

1. https://www.data.go.kr/data/15101609/openapi.do 에서 회원가입 → 활용신청 → `serviceKey` 발급
   → `.env`의 `CUSTOMS_API_KEY`에 등록 (GitHub Actions는 repo Secrets에 동일하게 등록)
2. 해당 페이지의 "활용가이드"/"참고문서"에서 실제 엔드포인트·파라미터명·응답 스키마 확인
3. `src/collectors/customs_trade.py`의 `CUSTOMS_BASE` 상수와 `_fetch_series()`/`_parse_items()`의
   파라미터명·필드명을 실제 문서에 맞게 수정
4. `python src/collectors/customs_trade.py` 로컬 실행 → `industry_indicators` 테이블에 데이터가
   쌓이는지, 값이 산업 규모에 맞는 합리적 범위인지(예: 반도체 수출 월 백억불대) 확인
5. 3장의 HS코드 중 특히 석유화학·디스플레이는 MOTIE 발표치와 비교해 범위가 맞는지 재확인 —
   필요시 HS코드를 여러 개 합산하도록 taxonomy 확장(`hs_code`를 배열로) 검토
6. 확인된 지표만 `industry_indicators.json`에서 `customs.verified: true`로 변경
7. `python src/signals/l0_industry.py` 재실행해 `l0_signals`에 `turning_up`/`rising` 등 실제
   상태가 산출되는지 확인 (지금까지는 데이터가 없어 전부 `관측부족`이었음)

## 5. 영향 범위

- `l0_industry.py`, `build_static.py`의 Turn Score(L0 가중 0.15), `timing.py`의 방향 판정
  (`_direction_up`의 L0 우선 조건)은 모두 이미 L0 데이터 존재를 전제로 작성돼 있어 코드 변경
  없이 실제 데이터만 채우면 즉시 작동한다.
- docs/05(시그널 보정 기획)의 "L0 성분 IC 검증"은 이 데이터가 몇 개월 쌓인 뒤에나 가능 —
  본 작업은 그 전제조건을 만족시키는 단계다.
